## 3. F2 — 배치 수준 국면 전환 (계획 밖 핵심 발견 후보)

같은 과제·같은 프롬프트·같은 모델 ID인데 **배치가 달라지면 행동 체제가
달라졌다**:

- **C(rename-sweep)**: 보정 배치(07-23 오전) 세션당 84~160초·1/3 성공
  → 본 배치(07-23 11:36~14:35) 435~929초·13/15 성공. 소요시간 4~6배,
  성공률 33%→87%. 전 세션 `modelUsage`가 `claude-sonnet-4-6`으로 동일 —
  기록된 모델 버전으로는 설명 불가.
- **D(orbit-propagator)**: 429 중단 전(#1~7, 07-23 14:35~15:56) 1/7 성공·
  평균 634초·17.6턴 → 재개 후(#8~15, 07-23 23:27~) 3/8 성공·839초·27.6턴.
  같은 방향(느리고 철저할수록 성공).
- **B(plugin-add)**: 보정 2/3 → 본 배치 15/15. 본 배치 안에서도 wall_s가
  쌍봉(3세션 ~200초 vs 12세션 ~700초) — "빠른 모드/철저한 모드"의 혼합.
- **A(buggy-pipeline)**: 대체로 빠른 모드(~180초)에 느린 세션 3개 산재.

해석: 실무 문헌은 서빙측이 배치 창 안에서 스냅숏·라우팅·용량 풀을 교체해
품질이 배치 중간에 이동하는 현상을 보고한다(스냅숏 로테이션). 우리 관측은
그것의 **행동-지표 수준 대응물**이고, 시간대·부하 교란과 분리되지 않으므로
인과 주장은 불가하다. 그러나 "유저가 통제할 수 없는 배치 수준 변동이
성공률을 33%↔87%로 흔든다"는 관측 자체가 세션 간 변동성 연구의 동기이자
발견이다. 재현 설계(같은 과제를 시간대를 바꿔 소배치 반복)가 본 실험의
자연스러운 arm 후보.

한계: C 보정 배치의 원자료는 보존하지 않았다(수치는 당시 세션 기록 기준).
**교훈: 이후 모든 배치의 원자료를 보존할 것.**

## 4. F3 — 조기 예측(RQ2)은 이 과제 세트에서 실패, 그리고 그 이유

AUROC@first-edit (성공 예측, 과제별):

| feature | C (실패 n=2) | D (실패 n=11) | pooled |
|---|---|---|---|
| cum_exploration | 0.94 | **0.33** | 0.56 |
| cum_files_read | 0.94 | 0.40 | 0.60 |
| cum_coverage | 0.92 | 0.44 | 0.62 |
| cum_errors / error_rate | 0.50 | 0.46 | 0.47 |

- 실패 표본이 실질적인 유일 과제 D에서 신호가 **역방향**이다: 실패 세션이
  첫 수정 전에 오히려 더 탐색했다(평균 4.5 vs 3.0). D의 성패는 수정 전
  탐색이 아니라 **수정 후 검증 루프**(성공 세션이 더 많은 호출 24.3 vs
  21.1, 더 긴 wall 811 vs 718초)에서 갈린다 — 숨은 오라클(수치 정밀도)
  과제의 구조적 특성. Majgaonkar의 "실패 실행의 72~81%도 문제 파일은
  찾았다"(coverage류 신호의 한계)와 정합.
- C는 0.92~0.94로 사전 등록 문턱을 넘지만 실패가 2건뿐이라 CI가 무의미.
  방향은 명확: C 실패 2건은 9개 파일 중 8개만 수정(1파일 누락)했고, 그중
  #11은 가장 빠른 축(474초)이었다 — "덜 탐색하고 빨리 끝낸 세션이 놓친다"
  는 C의 성패 기제는 RQ2 가설 그대로다.
- 함의: **RQ2는 과제 유형 의존적**이다. 스위프형(C)에서는 탐색-신호가
  작동하고, 숨은-오라클형(D)에서는 검증-신호(수정 후 행동)가 필요하다.
  "첫 수정 이전"이라는 사전 등록 시점 제약이 D 유형에서 원리적으로 불리.

## 5. F4·F5 — 위반: 천장 수정, 이상치 결합, H1b 미지지

- G2에서 "sonnet 21세션 누적 위반 0 → 준수 천장" 관측은 **n=60에서 수정**:
  위반 10건이 5세션(8.3%)에서 발생. 천장이 아니라 낮은 기저율.
- **위반 세션 5개 전부 해당 과제 평균보다 느렸고**(wall z-score +0.21,
  +0.23, +0.74, +1.75, +2.05), 그중 행동 이상치 2건(A#5 z+1.75, C#6
  z+2.05). 위반 세션 실패율 40%(2/5) vs 무위반 20%(11/55). n이 작아
  검정은 불가하나 "규칙 어기는 세션이 곧 행동 이탈 세션"(준수×일관성
  공분산) 방향과 일치 — 본 실험의 핵심 측정 지점으로 유지할 가치.
- H1b(의무형이 먼저 무너진다): 금지형 6 vs 의무형 4로 액면 미지지.
  의무형 4건은 전부 D#11 한 세션(같은 커밋 절차를 두 번 어김). 트리거
  기회 정규화를 해도 표본이 부족. 인접 발견: 2605.10039가 세션 내
  스텝별 준수 감쇠(스텝당 odds -5.6%)를 보고 — H1b의 "길이에 따라"
  부분은 세션 내 감쇠로 이미 관측된 셈이라 세션 간 축으로 재정위 필요.

## 6. F6 — diff 통계 (범위 준수)

- A·C: 30세션 전부 관련 파일 밖 수정 **0**. C 실패 2건은 깨끗한 시그니처
  (9개 중 8개 파일만 수정 — 파일 누락이 유일 실패 모드).
- B·D의 "관련 외" 항목은 전부 과제가 요구한 신규 파일(B: exporter 등록
  포함 3~4파일)이거나 `__pycache__` 잔재(D) — `relevant_files.txt`는
  읽기-커버리지 목록이지 수정 허용 목록이 아니므로 "불필요 수정" 라벨로
  쓰려면 과제별 편집 허용 목록이 따로 필요. 파일럿 결론: **범위 일탈
  (scope creep)은 이 과제 세트에서 관측되지 않음**. 도구 반영 시
  `__pycache__`/바이너리 필터 필요 (본 실험 전 처리).
